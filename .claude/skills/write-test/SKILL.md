# write-test

フレームワークに依存しない、テストを書くときの共通ルール。
フレームワークに依存するルールは以下のファイルを参照すること
- RSpecのテスト → `/write-rspec` スキル
- Vitestのテスト → `/write-vitest` スキル

## 基本原則

- 古典学派のテスト方針を採用する
- 公開された振る舞い（publicメソッド）を通じてテストする。実装の詳細を直接呼び出さない
- 各テストは独立して実行可能で、他のテストや実行順序に依存しない
- テスト対象クラスのテストに集中する。別クラスの振る舞いを網羅的にテストする必要はない
- AAA（Arrange-Act-Assert）パターンに従って構造化する
- テストコードに動的な期待値算出を持たせない（期待値はリテラルで記述する）
- **テスト内で if 文を使用しない**（条件分岐はテストケースの分割で表現する）
- テストを書き終えたら、他に追加・変更すべきテストがないか批判的に見直す
- 複数の分岐や複数のクラスに変更を加えた場合は、各分岐・各クラスにテストが揃っているか確認する

## テストダブルのガイドライン

- デフォルトは出力値ベーステスト。テストダブルの使用は最小限に
- 別クラスのメソッドのモックはできるだけ避ける。協調オブジェクトは実際に実行し、入出力の振る舞いで検証する
- ラップされた内部実装を直接モックしない
- モックすべき対象：外部API、外部サービス、キャッシュ
- モックしないもの：内部クラス/メソッド
  - データベースはフレームワークやライブラリによる
- モックを使う場合、テストの目的に応じてコミュニケーションベースと状態ベースのどちらかを選択する
  - コミュニケーションベース: 呼び出し回数と引数を検証する
  - 状態ベース: モック後のオブジェクトやデータの状態変化を検証する
  - 優先順位: 出力値ベース > 状態ベース > コミュニケーションベース
- **モックの戻り値を直接 assert しない**（自己充足的なテストになる）

## カバレッジ目標

- C0/C1 カバレッジの目安は 80%
- エッジケースとエラーケースをカバー
- 境界値分析、状態遷移テスト、デシジョンテーブルを活用
