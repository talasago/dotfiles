# update-docs

## 更新前の確認

- ドキュメントを追加・更新する前に既存ドキュメント（docs/配下、README、CLAUDE.mdなど）の内容を確認し、重複や矛盾が生じないようにする
- コード変更がドキュメントに記載済みの内容（設計・アーキテクチャ・テスト方針・挙動・利用者向け仕様など）に影響する場合、影響するすべての関連ドキュメントを同一PR内で更新する。ドキュメント更新は変更完了（Definition of Done）の一部である
- どのドキュメントが該当するかは変更内容から判断し、特定ファイルに限定して列挙しない

## コード参照（ファイルパス・コンポーネント名・関数名等）の更新

ファイル移動・リネーム・コンポーネント分割などのリファクタリングに伴いドキュメント内のコード参照を追従させる際、機械的な文字列置換（sedの一括置換など）だけで済ませない。

- 置換対象の文字列（例：`旧ファイル名`）を機械的に新しい値へ置き換える前に、その記述が指している機能が**現在どこに実装されているか**を実装コードで個別に確認する
- リファクタリングでは、単純にファイルが移動しただけでなく、機能が別のファイル・コンポーネントへ分割・移譲されていることがある。「旧パスの単純な書き換え」では正しい参照にならないケースがあることを前提に確認する
- 該当する実装がどこからも参照されなくなっていた場合（機能自体が不要になった、あるいは完全に置き換えられた場合）は、参照更新ではなくその記述・セクション自体の削除を検討する
- 更新後は、書き換えた参照先が実際にその機能を含んでいることを再度確認する（grep等で該当シンボル・処理が存在するか）
