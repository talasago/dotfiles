## 詳細ルールの参照トリガー
`.claude/rules/` には、既存ファイルを編集する際に自動で読み込まれる詳細ルールを配置。
ただし `.claude/rules/` の `paths` はファイルを**読むとき**にのみ発火し、コマンド実行のみの操作や新規ファイル作成では発火しない可能性がある。そのため、コマンド実行や「存在に気づく」ために必要な最小限の指示は以下に残す。

- テスト新規作成・修正時 → `/write-test` スキル 
  - 実装を変更した場合は、関連するテストの追加・修正が必要か確認する。
- GitHub Actions の `uses:` 追加・変更時 → `.github-actions.md`ルール
- Issue等が `claude.ai/design` のデザインURLを参照しているとき → `/design-mockup` スキル
- GitHub Issue を作成するとき → `/create-issue` スキル
- 依存ライブラリのインストール時 → `/install-dependencies` スキル
- パッケージマネージャーを使用してライブラリを更新するとき → `/dependency-management.md` ルール
- スキル新規作成時 → `.skills-management.md` ルール

## 着手前の調査・要件確認

### 要件が曖昧なissueへの着手方針

- Issue本文が空・または要件が曖昧な状態で着手する際は、実装レベルの設計（コンポーネント分割・ファイル構成等）に入る前に、対象ユーザー・スコープ・画面構成などの要件をユーザーと確定させることを優先する。要件確定前に詳細な実装計画は作らない。

## Subagentとメインチャットの使い分け

### Subagentを使うとき

- 該当タスクが、メインのコンテキストでは不要な冗長な出力を生成する場合。
- 特定のツールの制限や権限を適用したい場合
- このタスクは自己完結型であり、概要を返すことが可能である場合

具体例:

- コードベース全体を検索する
- 大量のファイルを読む
- ログ解析
- エラー調査
- ドキュメント調査
- 調査結果を要約して返す

### メインチャットでやり取りするとき

- 該当タスクには、頻繁なやり取りや反復的な改良が必要な場合。
- 計画、実装、テストなど、複数のフェーズが重要な共通点を共有している場合。
- 迅速かつ的を絞った変化を起こしている場合。
- レイテンシ重視。サブエージェントは新規に起動するため、コンテキストを収集するのに時間がかかる場合があるため。
- Plan mode下の設計フェーズでも、直前の調査で得た情報だけで実装計画を組み立てられる場合は、設計用のPlanサブエージェントを起動せずメインチャットで直接計画を書く。

### 調査依頼時の裏付けの明記

- 技術的主張（既知のバグ・仕様上の制約等）を含みうる調査は、裏付け確認ルールを組み込み済みの `research` サブエージェント（`.claude/agents/research.md`）に依頼する。
- 組み込みのExplore等に依頼する場合は、「主張の裏付けをWebSearch/WebFetch等で確認してから報告し、確認できない場合は未確認と明記する」ことを依頼プロンプトに明記する。明記を怠ると、報告に含まれる技術的主張の正しさは保証されない。

### 注意点

このセクションの基準がシステム側のガイダンスと矛盾する場合はユーザーに確認する


## Plan modeの計画ファイル

- 計画ファイルは、ユーザーから詳細な記述を求められない限り、コード例・背景説明（コンテキスト）を手短にまとめる。実装時に参照する具体的なファイルパス・関数名・設計判断の結論は残しつつ、冗長なコードスニペットや前提知識の再説明は避ける。

## 実装中のルール

### ビルド・テスト・Lintの運用

- 以下の場合は、ビルド・テスト・Lintを実行しない
  - ドキュメントやコードコメントのみの変更
  - Lintの自動修正などによる、動作に影響しない機械的な変更のみ
- コード（ロジック）を変更する場合、作業中は変更に関係するファイルのみを実行して都度確認する
- Lint設定を変更した場合は、設定の妥当性を確認するためLintを実行する
- プロジェクトで定義されたコマンドを確認して実行する
- フルテストとビルドはCIに任せるのでローカルでは実行しない
- ビルドまたはフルテストが2回連続で失敗した場合（エラーの種類を問わず）、自力修正を続けずにストップし、失敗内容と現状をユーザーに報告して判断を仰ぐ。

### ファイル削除の徹底

- ファイルを別ディレクトリへ移した場合、元のファイルがどこからも参照されていないことを確認する。
- 不要なら**その場で削除する**。
  - `git grep` や `grep -r` で参照がないことを確かめてから削除すること。
  - 移動と削除は同じコミットで行う。

### 関数・フックのシグネチャ変更時の網羅確認

関数・カスタムフックの引数を追加・削除・変更した場合、変更前から認識していたファイルの修正だけで完了とせず、**シンボル名で `src/` 全体を検索し、すべての呼び出し箇所を洗い出してから完了とする**。

### 変更を大部分元に戻す場合

- 実装済みの差分の大部分を取り消すことになった場合、ファイルを1行ずつ Edit で手戻りさせる前に、 `git diff <元ブランチ> -- <file>` で差分ゼロにできないか確認する。
- 差分ゼロ or 差分を最小限に抑えることができるなら`git checkout <元ブランチ> -- <file>` でファイルごと復元する
  - 無関係な既存バグ修正など、本当に残すべき差分が混在しているファイルのみ手動で編集する

### ドキュメント作成のルール

- ドキュメントを追加・更新する前に既存ドキュメント（docs/配下、README、CLAUDE.mdなど）の内容を確認し、重複や矛盾が生じないようにする
- コード変更がドキュメントに記載済みの内容（設計・アーキテクチャ・テスト方針・挙動・利用者向け仕様など）に影響する場合、影響するすべての関連ドキュメントを同一 PR 内で更新する。ドキュメント更新は変更完了（Definition of Done）の一部である。
- どのドキュメントが該当するかは変更内容から判断し、特定ファイルに限定して列挙しない。

### コードコメントの書き方

コード内コメントに、同一PR内でのレビュー対応（「レビュー指摘を受けて」「今回のPRで」等）中で完結する経緯を書かない。
そうした経緯はPR本文・コミットメッセージに書く。コードコメントに残すのは「なぜ今の実装がこうなっているか」という非自明な理由（隠れた制約・仕様上の落とし穴・回避策等）のみとする。

## GitHub 運用（PR・Issue）

### PR レビューコメントへの返信

- レビューコメントに対応したら、Github 側に**必ず返信する**。対応内容を1〜2行で書く。

### 複数仮説があるバグ修正時のルール

- バグ報告が原因の仮説を複数挙げている場合、実装した修正がどの仮説に対応するものかを PR 本文に明記する。
- 検証した他の仮説についても「採用しなかった理由」を明記する。
  - 1つの仮説の修正だけで「解決した」と暗黙に扱わず、他の仮説の検討過程を残す。

### PRの定期監視

PR作成後のCI失敗・レビューコメントへの反応は必要だが、**1時間おき等の定期チェックイン（send_later 等での自己再スケジュール）は行わない**。

CI失敗が既存テストの文言修正・単純なlintエラー等の軽微な範囲で直せる場合は、Plan modeで詳細な実装計画を都度作成せず直接修正してよい。

## Git運用
### PC上での場合
Claude code や codex などのweb版には適用しないルール
- commit / push はユーザーが明示的に指示した場合のみ実行する。指示がなければ絶対にやらない
  - 「コミットまでして」→ commit OK、push NG
  - 「pushして」→ push OK
  - 過去の承認は引き継がない。新しいコミット/pushのたびに都度指示を待つ
